## 1. プロジェクト概要
- 事業：**Alternative Process Lab / Ap2lab**（暗室ルームのレンタル等。湿板・サイアノタイプ等のオルタナティブ・プロセス写真）。
- 体制：少人数（3〜4名）。役割＝オーナー／経営パートナー／予約・顧客担当／作業担当。
- 暗室は**1室**運用予定。
- 予約・事前決済は **Square**（暗室の部屋料金を予約時に精算、店頭受付もあり）。

## 2. これまでの全体方針（重要な結論）
- **対外向け（顧客接点）と社内向け（業務管理）を分けて制作**する。
  - 社内向け業務管理 → **Glideで本番化予定**。現在は「叩き台」として動くHTML試作を `AP2管理WEB/ap2lab-manager.html` に作成済み（後述）。
  - 対外向けのコーポレート/予約導線 → **既存の Ap2lab サイト**を活用。
- **WordPress はセキュリティ運用負担（自前で守る必要）を理由に積極採用しない方針**。
  - 会員制・権限・コミュニティはWordPressの得意分野だが、脆弱性対応・更新・バックアップの自己管理が懸念。
  - 代替として、運営側がセキュリティを担う **マネージドなSaaS/ノーコード/BaaS** を優先。

## 3. 「顧客コミュニティ」の要件・論点（このセッションで出た話）
作りたいもの＝**顧客が参加できる会員制コミュニティ**。想定要素：
- **ID/パスワードのログイン**、**権限分け**（一般顧客／スタッフ等）。
- 顧客との**やりとり状況**の共有、**予約状況**の閲覧。
- **ワークショップ／セミナー／イベント**の告知・申込・**参加者管理**（内部アプリ側でも参加者リストを実装済み）。
- **Square予約／事前決済**との連動（暗室の部屋料金）。
- スマホの**ホーム画面アイコン（PWA）**で使えること。
- 顧客が多くなる前提（検索・索引などの使い勝手）。

### 内部アプリ（社内）と顧客コミュニティ（社外）の関係
- 社内 = AP2管理WEB（Glide想定）。経営数値・全予約・作業進捗など**内部情報**。
- 社外 = 顧客コミュニティ。**顧客が見える範囲だけ**（自分の予約・申込、イベント情報、告知、交流）。
- 二つは**データを一部共有**（例：予約・イベント参加者）するが、権限境界を厳密に分ける。

## 4. 技術選択肢（顧客コミュニティ向けの整理）
安全性（運営がセキュリティを担う順）：**既製SaaS ≧ ノーコード > BaaS自作 >> 自前WordPress**

| 方針 | 候補 | 向き |
|------|------|------|
| 既製コミュニティSaaS | Circle / Discord / LINEオープンチャット / Notion(共有) | 最速・最も安全。"自前アプリ感"は弱め |
| ノーコードで会員サイト | **Glide** / Softr / STUDIO(+会員) / Bubble | ログイン・権限・PWAが標準。Squareや予約連携も可 |
| BaaSで自作 | **Supabase**（認証・DB・RLS）＋ HTML/PWA | 自由度最大。運用に学習が必要だが自前WPより安全 |
| 予約＋決済 | **Square**（予約＋事前決済が一体）→ データを上記に連携 | 危険な決済処理は専門サービスに委譲 |

- 顧客コミュニティが「**会員制＋イベント申込＋予約閲覧**」中心なら、**Glide（会員アプリ）** か **Circle/Discord（交流主体）** が有力。
- "限りなく自分たちのアプリ"にしたいなら **Glide** か **Supabase＋PWA**。
- まず叩き台で形を見たい場合は、内部アプリと同じ要領で**単一HTMLのPWA試作**を作るのが早い（顧客向けにログイン＋自分の予約＋イベント申込＋お知らせ）。
