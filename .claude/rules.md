# Claude Code 作業ルール

このファイルは、Claude Codeでの作業時に必ず遵守するルールを定めています。
**すべてのルールは必ず実行すること。**

---

## 1. 日時・曜日の確認ルール

メール文章やドキュメント作成時に日付や曜日を記載する際は、以下のルールを厳守すること：

### 1.1 必須事項
- **必ずシステムカレンダーを確認する**
- `date`コマンドを使用して正確な日時・曜日を取得する
- 推測や記憶に頼らない

### 1.2 確認方法
```bash
# 現在の日時と曜日を確認
date "+%Y年%m月%d日 %A %H:%M"

# 特定の日付の曜日を確認（macOS）
date -j -f "%Y-%m-%d" "YYYY-MM-DD" "+%Y年%m月%d日 %A"
```

### 1.3 適用範囲
- メール文章の作成
- ドキュメントの作成・編集
- デイリーノートへの記録
- その他、日付や曜日を記載するすべての作業

### 1.4 実施タイミング
- 日付や曜日を記載する前に必ず確認
- 常に実際のシステム時刻を参照してから記載する

## 2. デイリーノート作成ルール

**作業中は必ずデイリーノートを作成および追記すること。**

### 2.1 デイリーノートの作成タイミング
- **作業開始時に必ず当日のデイリーノートを作成**
- 既に存在する場合は、そのファイルに追記
- 作業の区切りごとに追記
- 重要な作業を完了した時点で記録
- ユーザーからの指示や要求を受けた際に記録

### 2.1.1 会話要約の記録（重要）
会話履歴そのものは端末間で同期されない（PCローカル保存）。携帯のObsidian等から後日見返せるよう、**会話の要約をデイリーノートに残す**こと。

- **区切りごと**: 作業の区切りで、その時点までの要点を簡潔に追記する
- **セッション終了時**: 「終了します」と言われたら、変更ファイル一覧の前に、**そのセッション全体の要約**を書く
- **粒度は詳しめ**: 「何を話して何を決めたか」に加え、**そう決めた経緯・理由**も残す（後日「なぜそうしたか」まで追えるように）
- 記録フォーマット（2.4）の中で `## 会話要約` のような小見出しを使ってまとめる

### 2.2 デイリーノートの保存場所
- **パス**: `obsidian/DailyNotes/YYYY/MM/YYYY-MM-DD.md`
- **例**: `obsidian/DailyNotes/2026/01/2026-01-10.md`
- ディレクトリが存在しない場合は自動作成

### 2.3 記録する内容
- 実施した作業の内容
- 重要な決定事項
- 問題点や課題
- 参照したファイルやリソース
- 作成・更新したファイルのリスト
- その他、記録に値する情報

### 2.4 記録フォーマット
```markdown
# YYYY年MM月DD日 曜日

## 作業記録

### HH:MM - 作業タイトル

**環境:** Claude Code（このアプリ） / VS Code  ← どちらで作業したか
**ステータス:** ✅完了 / 🔄作業中 / ⏸中断  ← 再開時の判断用

**実施内容:**
1. 作業内容1
2. 作業内容2

**作成・更新ファイル:**
- [ファイル名](ファイルパス)

**決定事項:**
- 決定内容

**次にやること（あれば）:**
- 中断・継続する場合の次のステップ

---
```

### 2.5 注意事項
- 日付は必ずシステムカレンダーで確認してから使用
- 作業内容は簡潔かつ明確に記述
- 時系列で記録を追加
- ファイルパスはマークダウンリンク形式で記載
- **「環境」と「ステータス」は必ず記載する**（VS Code等の別環境との重複作業・同時編集を防ぐため）
- 作業を中断するときは「ステータス: 🔄作業中／⏸中断」と「次にやること」を必ず残す

## 3. ディレクトリ・ファイル管理ルール

### 3.1 ディレクトリ構造
```
obsidian/
├── DailyNotes/          # デイリーノート（YYYY/MM/YYYY-MM-DD.md）
├── チャットログ/         # プロジェクトごとのチャットログ
├── Clippings/           # 各種資料・ドキュメント
├── Projects/            # プロジェクト管理ファイル
└── Settings/            # 設定関連ファイル
```

### 3.2 チャットログの保存
- プロジェクトごとに `obsidian/チャットログ/[プロジェクト名]/` に保存
- ファイル名: `作業内容_日付.md`
- 例: `obsidian/チャットログ/Alternative Process Lab/修正ログ/事業骨子修正ログ_20260110.md`

### 3.3 ファイル作成時の注意
- 作成前にディレクトリの存在を確認
- 必要に応じて `mkdir -p` でディレクトリを作成
- 日付はシステムカレンダーで確認した正確な日付を使用

## 4. トークン管理ルール

### 4.1 トークン使用量の監視
- トークン使用量が**80%に達した時点**で、デイリーノートおよびチャットログを優先的に更新
- トークン使用量が**90%に達した時点**で、作業を一旦中断し、必ずログを保存

### 4.2 ログ保存の優先順位
1. デイリーノートの更新
2. プロジェクト関連のチャットログ保存
3. 重要な決定事項や問題点の記録

### 5.1 必須実行事項
以下のルールは**必ず実行**すること：

- [ ] トークン使用量90%で作業中断してログ保存
- [ ] プロジェクト関連の作業はチャットログに記録

このルールファイルは定期的に確認し、遵守状況をチェックすること。

## 6. 自動実行システム

### 6.1 実装済みの自動化機能

#### 会話開始時の自動実行（SessionStart）
- 会話開始時に自動的にデイリーノートを作成
- `.claude/hooks/conversation_start.sh` が自動実行される
- ディレクトリが存在しない場合は自動作成

#### 会話終了時の自動実行（SessionEnd）
「終了します」と伝えると自動的に：
- 作業サマリーを生成してデイリーノートに追記
- Git変更状況の統計を記録
- 今日のコミット履歴を記録
- セッション終了時刻を記録
- **Git自動コミット**（デフォルト有効）
  - すべての変更を自動ステージング＆コミット
  - タイムスタンプ付きコミットメッセージ
  - Claude Co-Authored-By付与

#### トークン使用量が高い時の自動実行（PreCompact）
トークンが高くなると自動的に：
- 作業サマリーをデイリーノートに保存
- Git変更状況を記録
- 警告メッセージを表示

### 6.2 利用可能なスクリプト

#### 作業サマリー生成
```bash
.claude/scripts/generate_work_summary.sh
```

#### Git自動コミット
```bash
.claude/scripts/auto_commit.sh [コミットメッセージ]
```

#### デイリーノート追記
```bash
.claude/scripts/add_daily_note.sh "作業タイトル" "作業内容"
```

## 7. 作業の進め方ルール

### 7.1 削除・上書き
- **新規ファイル作成・追記は自由に進めてよい**
- **既存ファイルの削除、内容の大幅な上書き（置き換え）は、実行前に必ず確認を取る**
- 作家活動の重要ファイルが多いため、取り返しのつかない操作を避ける

### 7.2 バージョンの扱い
- 文章（ステートメント等）を推敲・修正するとき、「新バージョンを作る（例: `_v10`→`_v11`、`V1`→`V2`）」か「同じファイルを上書き更新する」かは、**その都度確認する**
- 既存の命名習慣（`_v9/_v10`、`V1〜V4`）を尊重する

### 7.3 文体・トーン
- **基本は「です。」「ます。」調（敬体）**。文章はこれを既定とする
- ただし英文や、明らかに別の文体が適切なケース（作品コンセプト文で常体が映える等）は用途に応じて判断・確認する

### 7.4 確認の頻度
- 作業は**ステップごとにこまめに方向性を確認**しながら進める
- 調査・下書き・整理など可逆な作業はテンポよく進めてよい
- 削除・大幅上書きの前、バージョンの分岐点、文体の選択時は手を止めて確認する

### 7.5 セッション名・章タイトル
- **基本的に日本語**でつける
- 英語の方が適切な場合（固有名詞・製品名・技術略語・英文成果物名など）は無理に和訳しない

## 8. 締め切りタスク管理ルール

公募・応募やAlternative Process Lab等、**日付（締切）が決まっているタスク**は、デイリーノートとは別に一覧で一元管理する。

### 8.1 管理ファイル
- **ディレクトリ:** `obsidian/Clippings/締切タスク/`
- 活動領域ごとに3ファイルに分ける：
  - `締切タスク_写真家活動.md` … 写真家活動（公募・応募・展示・取材対応 等）
  - `締切タスク_AP2Lab.md` … Alternative Process Lab（略称 AP2Lab）
  - `締切タスク_その他.md` … 上記以外（コンサル・広報・技術メモ関連 等）
- 締切タスクが発生したら、内容に応じて該当ファイルへ振り分ける

### 8.2 運用
- 締切のあるタスクが発生・判明したら、該当ファイルの **## 進行中** に追記する
- **締切日が近い順**に並べる
- 各タスクは「ステータス / 内容 / 関連ファイル / 返答予定日 / メモ」を記載（フォーマットはファイル冒頭参照）
- ステータス: 🔲未着手 / 🔄対応中 / 📤提出済み（結果待ち）/ ✅完了 / ❌見送り
- 提出後も返答予定日まで追跡し、結果が確定したら **## アーカイブ** へ移動して実績として残す（締切直後に削除しない）
- 日付は必ずシステムカレンダーで確認してから記載する

## 9. アイデア記録ルール

外出先・就寝前など、**思いついた瞬間にアプリ（携帯含む）で記録するアイデア**を専用ディレクトリに残す。後日VS Code等で具現化する起点とする。

### 9.1 管理ディレクトリ
- **ディレクトリ:** `obsidian/Clippings/アイデア/`
- 運用ガイド: `obsidian/Clippings/アイデア/README_アイデア運用.md`

### 9.2 運用
- **トリガー: ユーザーが「アイデアです」と声掛けして始めたら、アイデア記録モードとして対応する**（以降の内容をアイデアとして扱う）
- ユーザーが「アイデアです」または「○○というアイデアを記録して」等と言ったら、**1アイデア＝1ファイル**で作成する
- ファイル名: `YYYY-MM-DD_アイデア要旨.md`（日付はシステムカレンダーで確認）
- 中身はREADMEのテンプレートに従う（思いついたこと / 背景 / 具現化メモ / 関連ファイル / 次にやること）
- ステータス: 💡思いつき / 🌱具体化検討中 / 🚧着手中 / ✅実現 / 🗄保留・見送り
- 思いついた時点では完璧でなくてよい。**まず残すことを最優先**する
- 携帯から記録された場合、frontmatterの「記録環境」に携帯/このアプリと明記する
- 締切が絡むアイデアは、セクション8の締切タスクにも登録する

### 9.3 Notion連携（3か所同時記録）
「アイデアです」で記録する際は、Obsidianだけでなく**Notionの2つのDBにも同時に登録**する（計3か所）。NotionはMCP連携で操作する。

1. **Obsidian** — `obsidian/Clippings/アイデア/` に1アイデア=1ファイル（上記9.2のとおり）
2. **Notion「アイデア帳」DB** — レコード追加
   - data_source_id: `045f6c12-bcda-4c84-a3aa-9452ca7635bf`
   - DB URL: https://app.notion.com/p/bcda84971a3f4e4e8950eb04eae6d809
   - プロパティ: アイデア(タイトル) / カテゴリ / ステータス / 優先度 / 思いついた日 / 詳細・メモ
   - カテゴリ候補: 撮影企画 / ロケ地 / 構図・技法 / 機材 / 作品制作 / 発信・SNS / その他
3. **Notion「撮影・活動カレンダー」DB** — 思いついた日にレコード追加（カレンダーに乗せる）
   - data_source_id: `89d947dc-f555-4fbe-b0bf-4cae81320bf5`
   - DB URL: https://app.notion.com/p/236751c0f68a47bdb21243bde5946542
   - 設定: 種別=「アイデア」、日付=思いついた日、タイトル=アイデア要旨、記録・メモに概要
   - これにより「思いついた日」がカレンダー表示に反映される
```bash
.claude/scripts/save_chat_log.sh "プロジェクト名" "作業タイトル" "チャット内容"
```

#### トークン使用量チェック
```bash
.claude/scripts/check_token_usage.sh <現在のトークン数> <最大トークン数>
```

### 6.3 環境変数設定

`.claude/settings.local.json` の `env` セクションで設定：

- `AUTO_COMMIT_ON_SESSION_END`: セッション終了時の自動コミット（デフォルト: `true`）
- `AUTO_PUSH`: コミット後の自動プッシュ（デフォルト: `false`）
